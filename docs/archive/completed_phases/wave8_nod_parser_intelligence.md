# Wave 8 — NOD Parser: Two-List Intelligence Output

Status: complete
Builds on: Wave 7.

The NOD parser produced only 3 keyterms and misattributed the signing
firm. This wave makes it generate two clearly-separated outputs — a
Deepgram speech-intelligence list and a UFM legal-metadata list — and
fixes the extraction bugs found against the real Brain & Spine /
Heath Thomas notice.

## The two lists

The parser response now separates concerns:

**Deepgram intelligence** — `keyterms`, `speaker_hints`, `deepgram_config`.
Speech-recognition priming: what to feed Deepgram so it transcribes
names, firms, and legal phrases correctly.

**UFM metadata** — `fields`, `metadata`, `appearances`. Legal-document
fields for the Texas UFM template.

These are generated by separate code paths (`intelligence.py` vs the
field extractors) so ASR logic and legal-formatting logic never couple.

## Keyterms: 3 -> 28, categorized

Before, `keyterms` held only the deponent, the signing attorney, and one
firm. Now `backend/services/nod_parser/intelligence.py` builds a
comprehensive, de-duplicated, categorized list. Against the test NOD:

- **Person (8)** — deponent, plaintiff, the individual co-defendant, all
  four attorneys, and the scheduler.
- **Law Firm (3)** — Cukjati Law Firm, Brain & Spine, Brothers Alvarado.
- **Organization (2)** — Home Depot U.S.A., Inc. and the a/k/a alias.
- **Geographic (4)** — cities + federal district + division.
- **Case Identifier (1)** — the civil action number.
- **Legal Term (10)** — a conservative curated set of deposition-specific
  phrases.

Each keyterm carries a `priority` (1-10) used for display ordering and to
decide which terms survive the 100-term cap.

## Speaker hints and Deepgram config

- `speaker_hints` — `{name, role}` for the witness and every attorney
  (`witness` / `plaintiff_attorney` / `defense_attorney`). This feeds the
  Workspace's speaker-labeling step (the Q/A labeling limitation flagged
  in Wave 7).
- `deepgram_config` — recommended batch request settings, consistent with
  `backend/deepgram/client.py` (`diarize_model: latest`, `nova-3`, etc.).

## UFM field bugs fixed

- **Signing firm misattribution.** A NOD signed by co-counsel from two
  firms lists both. The old code took the *first* firm after
  "Respectfully submitted" — here Cukjati Law Firm — when the signer,
  Steven A. Nunez, belongs to Brain & Spine. The firm is now read as the
  caps block immediately above the `/s/` signature, handling firm names
  split across multiple lines.
- **Incomplete court name.** `ufmCourt` was just "Western District of
  Texas". It is now the full "United States District Court, Western
  District of Texas, San Antonio Division". The old division regex
  spanned the multi-line court header and self-rejected; a line-anchored
  extractor replaces it.
- **All attorney appearances.** New `extract_appearances()` mines every
  attorney from the signature block ("State Bar No." pattern) and the
  "TO: ... attorney of record" line, each with name, firm, bar number,
  and side (plaintiff/defendant).

## What was adopted from the target spec, and what was not

The target spec proposed the two-list separation — a sound architectural
call, adopted here. Three of its other suggestions were **not** built,
deliberately:

- **Phonetic variants** (e.g. "Cukjati" -> "Cook-jotty"). Deepgram Nova-3
  Keyterm Prompting takes correctly-spelled terms; it does not consume a
  separate phonetic-spelling input. Feeding invented pronunciations would
  add noise, not accuracy. The correct spelling as a keyterm is what
  helps.
- **Per-term Deepgram weighting.** Nova-3 Keyterm Prompting accepts a
  plain list of terms with no per-term boost (only the legacy Keywords
  feature did). The parser keeps a `priority` value, but it is used only
  for our own ordering and cap-survival — it is not sent to Deepgram, and
  the code/comments say so.
- **"Expected speaker vocabulary"** (generic phrases like "objection",
  "I don't recall"). These are common words Deepgram already transcribes
  correctly; as keyterms they would dilute the list.

The spec's own example also misread the documents — it labeled the
deponent ("Heath Thomas") as the court reporter and the plaintiff's
attorney as the reporter. The parser extracts these correctly.

## Frontend

The enriched `keyterms` and corrected `fields` flow into the existing
Stage 1 UI automatically (keyterms table + UFM form). The new
`speaker_hints`, `deepgram_config`, and `appearances` are in the API
response and ready for downstream stages; surfacing them in dedicated UI
panels is a follow-up.
